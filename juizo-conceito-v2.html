02 · Modelo de Versionamento

O processo como
repositório Git

O processo não é uma pilha linear de documentos — é um DAG (Directed Acyclic Graph), exatamente como o modelo interno do Git. O main branch é o processo principal. Incidentes são branches. Deferimentos são merges. O trânsito em julgado é o tag de release.

03 · Princípio da Previsibilidade

Domando a entropia
processual

Toda tecnologia é uma tentativa de domar a entropia. O processo judicial deveria ser isso — e virou o oposto. A lógica do "pedir não custa" é um exploit do sistema: custo zero para inserir ruído, custo alto para processar. Quem paga é o juiz, o cartório, a parte contrária.

Fundamento · Previsibilidade como design · Ruído tem custo para quem o gera

Sistema Atual
Custo zero para inserir ruído no processo. Chicanas, retórica vazia, pedidos descabidos — tudo aceito, tudo numerado, tudo abarrota o main.
Regra Juízo
Pedido fora da gramática do estado atual → isolado como branch. Se indeferido → nó morto no DAG. Não contamina o main. Auditável, mas sem peso.
Resultado
O main do processo reflete apenas o que importa. Análise do juiz é cirúrgica. Tribunais superiores lêem o DAG limpo + branches para contexto.
FSM GARANTE: atos fora de fase → rejeitados prazos → computados automaticamente nulidades → impossíveis por design aleatoriedade estrutural → eliminada discricionariedade legítima → preservada
04 · Máquina de Estados

O processo como
máquina de estados

Cada estado tem transições permitidas e transições barradas. O sistema não aceita atos fora de fase — eliminando nulidades na origem. O Judiciário é o servidor. Os atores são clientes. Todo ato é um POST semântico validado.

05 · Linguagem Processual

juizo-core
a DSL do processo

Peças processuais não são documentos — são objetos de dados validáveis. Uma biblioteca Python com sua própria gramática: se o sistema não compreende, retorna erro. Fim da inépcia, fim do esquecimento de campos obrigatórios, fim do PDF digitalizado. 

from juizo import Processo, Ato, Parte, Testemunha, Pedido, Fundamento, Rito

# O processo como objeto computável
processo = Processo(
    rito=Rito.JUIZADO_ESPECIAL_CIVEL,
    vara=Vara("1JEC-CURITIBA-PR")
)

# A petição inicial como objeto estruturado
inicial = Ato.PETICAO_INICIAL(
    processo=processo,
    autor=Parte(cpf="xxx.xxx.xxx-xx", qualificacao={...}),
    reu=Parte(cnpj="xx.xxx.xxx/xxxx-xx"),

    fatos="...",           # campo livre · obrigatório
    argumentacao="...",   # campo retórico · opcional

    # sistema sugere com base nos fatos; advogado confirma
    fundamentacao=[
        Fundamento.CDC_ART_14,
        Fundamento.DANO_MORAL_CONFIGURADO,
    ],

    pedidos=[
        Pedido(tipo=Pedido.CONDENACAO_PECUNIARIA, valor=5000.00),
        Pedido(tipo=Pedido.DANO_MORAL, valor=10000.00),
        Pedido(tipo=Pedido.TUTELA_ANTECIPADA, urgencia=True),
    ],

    # testemunha sem endereço → comparece independente
    # sistema NÃO expedirá mandado / carta de intimação
    testemunhas=[
        Testemunha(
            nome="Maria Silva",
            cpf="yyy.yyy.yyy-yy",
            endereco=Endereco(rua="...", cidade="Curitiba"),
            comparece_independente=False  # sistema expede intimação
        ),
    ],
)

# Validação ANTES do protocolo
resultado = inicial.validar()

if resultado.erros:
    # lista exata do que falta · sem surpresas
    raise ErroProtocolo(resultado.erros)

# Protocola → dispara FSM → notifica → computa prazos
processo.protocolar(inicial)
# Pedido incidental → branch automático
incidente = Ato.PEDIDO_INCIDENTAL(
    processo=processo,
    tipo=Incidente.ADITAMENTO_TESTEMUNHAS,
    requerente=advogado,
    fundamentacao="substabelecimento + novos fatos",
    novas_testemunhas=[...],
)

# Cria branch · não toca no main
branch = processo.abrir_incidente(incidente)
# branch.id = INC-2024-00847
# branch.ref_main = PROC-2024-11234

# Juiz defere → merge automático
# Juiz indefere → branch morto
# Ambos: auditáveis, rastreáveis
# Ferramentas de migração do legado
from juizo.migration import LegacyImporter

importer = LegacyImporter(source="projudi")

# Pipeline de ingestão de legado
importer.from_pdf(arquivo)        # OCR → objeto
importer.from_video(video)        # transcrição → texto
importer.from_scan(imagem)        # visão computacional
importer.from_manuscrito(foto)    # HTR handwriting

# IA classifica → serventuário valida
# 70-80% automatizado, 20-30% humano
06 · Inteligência Jurisdicional

O Agente IA
do magistrado

Cada magistrado terá um Agente IA que aprende seus posicionamentos, entende seu estilo de fundamentação, conhece sua jurisprudência pessoal. Não é automação da decisão — é o assessor digital perfeito.

Camada 1 · Assessor Digital
Clone do Magistrado
Aprende 300+ casos do juiz. Quando entra processo novo: pré-analisa, sugere fundamentação provável, identifica padrão de julgamento. O juiz confirma, ajusta ou diverge — mas o trabalho pesado já foi feito.
Camada 2 · Previsibilidade para as Partes
Litigar com Informação
Advogados e partes terão acesso a probabilidade de êxito baseada no histórico do juiz sorteado. Filtro de litígios sem fundamento. Acordos tornam-se matematicamente racionais antes do processo iniciar.
Camada 3 · Agregação Nacional
Jurisprudência Emergente
Posicionamentos de N magistrados agregados em tempo real. O entendimento majoritário de fato — não o de 7 ministros. A pressão da base sobre o topo. Ordem emergente a partir do comportamento local.
Camada 4 · Auditoria de Equidade
Viés Visível, Não Censurado
Se o modelo aprende viés histórico, o sistema o torna visível e questionável — não o corrige silenciosamente. Transparência sobre o comportamento real do Judiciário. Pressão por uniformidade onde houver injustiça.

Pela primeira vez na história, a fixação do "entendimento" judicial sobre um caso virá de baixo. Não será o entendimento de sete pessoas — será o entendimento majoritário de fato, que deverá viger.

Visão · Jurisprudência como dado agregado · Democracia jurisdicional

07 · Arquitetura Funcional

Os pilares do
novo sistema

01
FSM por Rito
Máquina de estados para cada rito processual. Transições válidas bloqueiam nulidades na origem. O sistema é estruturalmente correto — não por vigilância, mas por design.
02
DSL Processual
juizo-core: biblioteca Python com gramática própria. Peças são objetos. Validação antes do protocolo. Fim da inépcia, do esquecimento de campos, das peças escaneadas.
03
Git Jurisdicional
Processo como DAG. Incidentes como branches. Deferimentos como merges. Trânsito em julgado como tag de release. Histórico imutável com hash criptográfico em cada commit.
04
Deliberação Computável
Despacho do juiz dispara atos automáticos: ofícios, mandados, intimações, prazos. Linguagem natural do juiz extraída por IA e convertida em ações sistêmicas.
05
Agente IA por Magistrado
Clone jurisdicional que aprende cada juiz. Assessor digital. Previsibilidade para as partes. Subsídio para jurisprudência emergente de baixo.
06
Integração Total
Tribunais, delegacias, penitenciárias, bancos, acolhimentos — um dataset único. REST semântico. Cada ator é um cliente do servidor jurisdicional.
08 · Estratégia de Implantação

Strangler fig:
migração sem ruptura

O novo sistema cresce em volta do antigo até que o antigo murche por si só. Duas opções para os tribunais — sem obrigação de escolha imediata.

Opção A · Parallel Run

  • Processos atuais continuam no sistema legado (PJe, Projudi, e-SAJ) até arquivamento natural
  • Novos processos entram no Juízo desde o primeiro dia
  • Zero risco operacional para o tribunal
  • Extinção orgânica do legado em 3–7 anos dependendo do acervo
  • Recomendado para início de implantação

Opção B · Importação Total

  • Pipeline de ingestão: OCR, transcrição de vídeo, HTR manuscrito
  • IA pré-classifica 70-80% dos atos — serventuário valida e corrige
  • Cada processo importado recebe schema completo no novo formato
  • Trabalho intenso mas único — amortizado pela qualidade futura
  • Recomendado apenas após maturidade do sistema no Opção A
